# バグ調査・修正のワークフロー

## バグ修正前に必ず原因と修正案を提示する

バグの調査・修正を行う際は、**実装の前に必ず**以下を提示すること。

1. **根本原因** — なぜそのバグが起きているか（コードの該当箇所を明示）
2. **修正方針** — どう直すか（トレードオフがある場合はそれも示す）

提示後、ユーザーの合意を得てから実装に入る。

```
❌ 調査 → 即実装 → 後から説明
✅ 調査 → 原因+修正案を提示 → 合意を得て実装
```

## なぜこのルールか

バグ修正は原因の見立てが誤っていると、正しい箇所を直せない。
また、修正方針によってはアーキテクチャや仕様に影響するため、ユーザーが判断すべき選択肢を含む場合がある。
先に提示することでそのコストを最小化できる。

## 適用範囲

- 既存機能のバグ修正
- 「保存されない」「動かない」「エラーになる」などの不具合対応

## 適用しない場面

- ユーザーが明示的に「直して」と指示済みで原因が自明な場合
- タイポ修正・lint エラーなど説明不要な軽微な修正
